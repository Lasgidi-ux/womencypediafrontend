# Plan: Remove Static Content and Use Dynamic Strapi Data

## Problem Analysis
The frontend currently has static fallback data for biographies, collections, and features when Strapi API calls fail. The user wants to eliminate all static content and ensure all data comes dynamically from Strapi CMS.

## Current State
- Static data exists in `js/data.js` with biographies, categories, regions, eras, featuredCollections
- Multiple scripts have fallback logic to static data when API fails
- Some scripts already load dynamic content but fall back to static
- Homepage loads dynamic content but falls back to static HTML

## Goals
1. Remove all static biography, collection, and feature data
2. Update all scripts to use only dynamic Strapi data
3. Show appropriate loading/error states instead of static fallbacks
4. Ensure consistent dynamic content loading across all pages

## Implementation Plan

### Phase 1: Data Removal
- Remove `biographies`, `featuredCollections` from `js/data.js`
- Keep only necessary static data (categories, regions, eras) if used for filtering
- Update any references to removed static data

### Phase 2: Script Updates
- Update `js/browse-logic.js`: Remove `loadStaticEntries()` and `loadStaticFilterOptions()` fallbacks
- Update `js/collections-inline-1.js`: Remove `fallbackCollections` and static fallback logic
- Update `js/featured-inline-1.js`: Ensure it only shows loading/error if no data from API
- Update `js/homepage.js`: Remove static HTML fallbacks, show loading states
- Update other scripts using static data to require Strapi

### Phase 3: Error Handling
- Implement proper loading states and error messages
- Add user-friendly messages when content is unavailable
- Ensure graceful degradation without static content

### Phase 4: Testing
- Verify all pages load content from Strapi
- Test error scenarios (API down, no content)
- Ensure no references to removed static data remain

## Files to Modify
- `js/data.js` - Remove static data
- `js/browse-logic.js` - Remove static fallbacks
- `js/collections-inline-1.js` - Remove static collections
- `js/featured-inline-1.js` - Update error handling
- `js/homepage.js` - Update fallback behavior
- Other JS files using static data

## Strapi Content Types Verification
Ensure these content types exist and have appropriate permissions:
- Biography (with featured field)
- Collection (with featured field)
- Homepage (single type)
- Education Module
- Teaching Resource
- Leader
- And others as needed

## Risk Assessment
- Content may not load if Strapi is down or misconfigured
- Users will see loading/error states instead of content
- Need to ensure Strapi has all required content published
